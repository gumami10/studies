1. Collapse bookmark + star stores into one Placemark module

Strong local-substitutable
src/stores/bookmarks.ts src/stores/starred.ts src/router/index.ts src/pages/StarredPage.vue src/components/ui/StarButton.vue src/components/ui/BookmarkButton.vue src/components/ui/MobileToolbar.vue

Before: two shallow stores

bookmark store

toggle()1 call site
count0 call sites
leaks: RouteName→source map duplicated
starred store
isStarred()3 call sites
toggle()2 call sites

leaks: DOM clone on star, RouteName→source map duplicated

After: one deep Placemark module

Placemark store
place(knowledgeId, sectionId, title, html)
remove(id)
isPlaced(id)
bySource
latestByModule
single seam · 5 call sites · 1 test file

interface shrinks from 18 methods → 5; implementation absorbs DOM clone + source resolution + scroll logic

Problem

Two Pinia stores solve the same problem (save-for-later on content sections) with different interfaces, different storage keys, and no shared logic. Source identification and DOM cloning leak into UI components.

Solution

Merge both stores into one Placemark store. The interface distills to five methods. The implementation absorbs the DOM clone operation, the source resolution (reading from the knowledge manifest, not brittle route-name matching), and the scroll-to logic currently in the router's scrollBehavior.

Wins

  • locality: bookmarks, stars, auto-scroll, source resolution co-locate in one module
  • leverage: one interface, two UI components (Button + StarredPage) instead of 4+
  • interface shrinks: 18 exported methods → 5
  • delete 2 store files, deduplicated RouteName→source maps in 3 files
  • tests concentrate: one test file instead of two testing overlapping concerns

2. Remove brittle RouteName→source maps — read from manifest

Worth exploring in-process
src/components/ui/StarButton.vue:51-56 src/components/ui/MobileToolbar.vue:85-101 src/router/index.ts:47-74

Before: source maps duplicated in 3 files

StarButton: route.name === 'chapters' ? 'CTAL-AT' : route.name === 'metrics' ? 'Quality Metrics' : ...
MobileToolbar: route.name === 'chapters' ? 'CTAL-AT' : route.name === 'metrics' ? ... : route.name === 'tae' ? 'CTAL-TAE' : ...
+ grows with every new knowledge module

After: one query on the manifest

manifest.navLabel
← auto-resolved from route.meta.knowledgeId

add a `getLabel(knowledgeId: string): string` to the catalog, source every caller from it

Problem

Two files contain near-identical if/then/else chains mapping route names to human-readable source labels. Adding a knowledge module means updating both chains. The manifest already carries navLabel and title.

Solution

Use route.meta.knowledgeId (already populated by the router) to look up the source label from the catalog. The manifest's navLabel is exactly the string these chains reconstruct.

Wins

  • locality: source label lives in one place — the manifest
  • leverage: catalog interface already exists; one new lookup method serves 3 call sites
  • adding a knowledge module never touches these files again

3. Bookmark at section-level, not chapter-level

Strong in-process
src/components/ui/BookmarkButton.vue src/pages/ContentPage.vue src/composables/useSectionTracker.ts src/stores/bookmarks.ts

Before: bookmarks stop at chapter boundary

module: Chapters
Chapter 1 📌 bookmarkable
Section 1.1 — not bookmarkable
Section 1.2 — not bookmarkable
Chapter 2 📌 bookmarkable
Section 2.1 — not bookmarkable

bookmarks stop here; sections below this level have no bookmark. Stars are the only option for sub-sections.

After: bookmarks and stars unify at section granularity

Placemark module (one interface)
Chapter 1 📍 placeable
Section 1.1 📍 placeable
Section 1.2 📍 placeable
Chapter 2 📍 placeable
Section 2.1 📍 placeable

same interface for chapters, sections, sub-sections; auto-scroll works at every level

Problem

The BookmarkButton receives a sectionId at the chapter level only. Within a chapter, all its internal sections (ContentSection blocks) are only starrable via the StarButton, which uses active scroll position — not the chapter it's visually attached to. A user scrolling through "Section 2.3" clicks the star button near "Chapter 2" heading and stars Section 2.3. This is the most confusing interaction in the app.

Solution

Attach the placemark button to every section, not just every chapter. The section id and section title are pulled from the section's DOM or content block id. The auto-scroll in the router reads the latest placemark for the knowledge module and scrolls to it — at any granularity. This naturally absorbs candidate 1.

Wins

  • leverage: one button component that works for chapters and sections alike
  • locality: auto-scroll reads from the same store that creates placemarks
  • no more "what am I starring?" confusion — button is on the section it stars

After: one deep toggle module

usePlacemarkToggle(sectionId, sourceId) composable
→ Returns: { isPlaced, toggle(), html }
→ Used by: StarButton, MobileToolbar

one implementation, two adapters (desktop button, mobile toolbar button)

Problem

The entire star toggle operation (clone DOM, remove buttons, determine source label, store.toggle) exists in two places. The MobileToolbar version also removes an extra CSS class. When the logic changes, both must be updated in sync.

Solution

Extract a usePlacemarkToggle composable that takes a section id and knowledge manifest id, and returns reactive state + toggle function. Both the desktop StarButton and MobileToolbar adapters consume it.

Wins

  • locality: toggle logic lives in one composable
  • leverage: two adapters justify the seam
  • test the composable once; both adapters trust it

5. Make the starred page navigable — each placemark links back to its source context

Strong in-process
src/pages/StarredPage.vue src/stores/starred.ts src/router/index.ts src/types.ts

Before: StarredItem has no navigation target 

StarredItem {
  id: "sec-1"      // section DOM id
  title: "My Sec"
  source: "CTAL-AT" // brittle label
  html: "<div>..."  // frozen DOM snapshot
  // no knowledgeId — no way to navigate
}

StarredItem can't know which knowledge module it came from → can't build a link to its source page.

After: Placemark carries knowledgeId 

Placemark {
  id: "sec-1"
  title: "My Section"
  knowledgeId: "chapters"
  sectionId: "sec-1"      // for scroll-to
  html: "<div>..."
}
// Router.buildLink(knowledgeId, sectionId)
// → /chapters#sec-1

every placemark resolves to a navigable URL

Problem

The StarredPage renders saved HTML fragments grouped by source label, but there is no link back to the source knowledge module. A user staring at a starred section must navigate back manually. Worse, the StarredItem type has no knowledgeId field — only a brittle human-readable source label.

Solution

Add knowledgeId to the placemark type. On the starred page, render each group with a link to the source knowledge module path, optionally including a #sectionId hash so the router's scrollBehavior scrolls to the exact section. The source label remains for display only; the navigation target is the machine-readable knowledgeId.

Wins

  • leverage: one placemark, two surfaces (starred page + scroll behavior)
  • locality: navigation target stored once, consumed by router + StarredPage
  • deletion test: removing source label is safe; navigation just needs knowledgeId

6. Replace DOM-clone persistence with structured content refs

Worth exploring local-substitutable
src/stores/starred.ts src/pages/StarredPage.vue src/components/content/ContentRenderer.vue src/components/content/ContentSection.vue

Before: frozen HTML snapshots break when DOM changes

Star: section.cloneNode(true).outerHTML
Store: StarredItem.html = "<section id=...>...</section>"
Render: v-html="item.html" → brittle
Fails when: CSS class names change, heading levels change, PrimeVue markup differs across versions, content refactor changes DOM structure — stale HTML stored in localStorage.

After: store a content reference, re-render from data

Star: { knowledgeId, sectionId, type: 'section' }
Store: Placemark.contentRef = { knowledgeId, sectionId }
Render: lookup section from ChapterData → render via ContentRenderer

same ContentRenderer that renders live pages renders the starred page; HTML never stored

Problem

The starred store persists serialized DOM HTML (outerHTML) to localStorage. On the StarredPage, this HTML is rendered via v-html and further manipulated with native DOM APIs (attachStarButtons). This bypasses Vue's component system entirely. Any change to CSS, heading levels, or component markup silently breaks stored starred sections.

Solution

Instead of storing HTML, store a lightweight content reference: { knowledgeId, sectionId }. The StarredPage uses the same ContentRenderer that renders the live ContentPage — just re-render the referenced section from its data module. The "source" label is resolved from the manifest. Storage shrinks from kilobytes of frozen HTML to a few bytes of structured ref.

Wins

  • locality: one rendering path (ContentRenderer) for both live and saved content
  • leverage: re-rendering from data means all design changes apply to saved content automatically
  • storage size drops >90% — no more HTML bloat in localStorage
  • no DOM manipulation in StarredPage — pure Vue
  • e2e tests survive CSS refactors
Note: Requires the section content to be addressable by its id from the ChapterData structure. Current ChapterData stores content as flat arrays; referencing a section by id requires an index or recursive lookup. This is a structural prerequisite — worth scoping as a v2 of the placemark refactor.

Top recommendation

Start with Candidate 1: Unify stores into Placemark

The two Pinia stores (bookmarks + starred) share identical infrastructure (localStorage persistence, Pinia setup, toggle pattern) and nearly identical purpose (save-for-later on content). Their seams leak across the codebase — the source label resolution is duplicated, the auto-scroll behavior spans router + store + page, and the DOM-clone logic exists in two places. Candidate 1 is the foundation block: merge the stores, then build candidates 3 (section-level granularity), 4 (toggle composable), and 5 (navigable starred page) on top of it. Candidate 6 (structured refs instead of HTML) is the next natural step but has a structural prerequisite — index the ChapterData sections by id.
